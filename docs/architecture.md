# Architecture

Flowmail is a single-organization Cloudflare-native application.

```text
Browser
  -> React Router UI
  -> Hono Worker API
  -> D1: product, contacts, campaigns, events, inbox
  -> R2: CSV imports, raw inbound email
  -> Queues: campaign sends
  -> Workers AI: draft generation
  -> Email Service: outbound email
  -> Email Routing: inbound replies
```

## Main flows

- Setup checks bindings, sender-domain readiness, and manual Cloudflare steps.
- Setup can send a test email through the current product sender, save an encrypted Cloudflare API token, inspect zone MX and Email Routing state, and create/update the configured reply-to route to the Worker.
- CSV import parses, validates, dedupes, stores source metadata, and writes an import report.
- Manual suppression entries immediately mark matching contacts as unsubscribed and exclude them from sends.
- Campaign drafts are generated by Workers AI when available and by deterministic fallback when unavailable.
- Campaign approval runs compliance checks, creates recipient rows, and enqueues send jobs.
- Send jobs re-check suppression and idempotency before using Email Service.
- Click, unsubscribe, and session tokens are signed with an automatically generated instance secret unless `TRACKING_SECRET` is provided as an override.
- Inbound email is stored in R2, attributed by Flowmail headers, message-id references, or latest sent recipient fallback, summarized in D1, classified, and converted into an Agent draft action.
- Agent reply drafts are sent only after operator confirmation and are logged as events.

## Trust boundary

Flowmail starts with a built-in admin password login. v0.1 does not implement per-user RBAC.
